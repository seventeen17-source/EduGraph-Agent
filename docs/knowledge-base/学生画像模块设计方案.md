# 学生画像模块设计方案

## 一、画像在系统闭环中的位置

```text
┌─────────────────────────────────────────────────────┐
│                  学习闭环                            │
│                                                      │
│  对话构建画像 → 知识诊断 → GraphRAG → 资源生成        │
│       ↑                                      ↓        │
│       └── 画像更新 ← 学习评估 ← 练习测试 ←─┘        │
└─────────────────────────────────────────────────────┘
```

画像不是一次性填完就搁置的静态表单，而是**贯穿整个学习闭环的活数据**：

- **闭环入口**：对话构建初始画像，决定第一条学习路径和第一批资源推荐
- **闭环中间**：GraphRAG 用画像做个性化检索排序，资源生成 Agent 用画像调整内容
- **闭环末端**：练习评估结果反向写回画像，触发路径重规划

所以画像模块的设计不只是一个 CRUD API，它要回答：

1. **怎么收集**——对话式构建 + LLM 抽取
2. **怎么存储**——结构化持久化 + 更新历史追溯
3. **怎么交付**——前端画像卡片展示，每轮对话实时刷新
4. **怎么被消费**——GraphRAG、路径规划、资源生成都读它
5. **怎么更新**——答题结果、辅导反馈、手动修改都能触发
6. **怎么解释**——每次更新都带来源、置信度、时间戳

## 二、8 维画像的完整数据模型

当前系统中只有一个 `StudentProfileInput`（4 个扁平字段：`weak_points`、`preferences`、`goal`、`mastery`），它只是一个 API 入参，不能支撑完整的画像生命周期。这里不建议直接拆掉现有结构，而是新增完整 `StudentProfile`，并保留 `StudentProfileInput` 作为 GraphRAG 和资源生成接口的轻量兼容适配器。这样已有 `/api/graphrag/query`、`/api/agents/generate-resources` 可以平滑迁移，不会因为画像模块上线而破坏当前已跑通的 demo 链路。

### 2.1 维度总览

| 维度 | 字段名 | 优先级 | 含义 | 谁来填 |
|---|---|---|---|---|
| 专业背景 | `background` | 必问 | 学生所在的专业、年级、学校类型、已修课程 | 对话首轮自然语言提取 |
| 学习目标 | `learning_goal` | 必问 | 为什么学这门课、目标类型、期望达成的结果 | 对话首轮提取 |
| 知识基础 | `knowledge_base` | 必问 | 已知哪些主题、不熟悉哪些主题、各主题掌握程度 | 对话首轮提取 + 后续测验微调 |
| 认知风格 | `cognitive_style` | 重要 | 偏理论、偏案例、偏代码还是偏图解 | 对话追问提取 |
| 学习偏好 | `preferences` | 重要 | 喜欢什么资源类型、单次学习时长、难度偏好 | 对话追问提取 |
| 易错点 | `weak_points` | 可推断 | 自述薄弱点 + 系统通过答题诊断的薄弱点 | 对话提取 + 答题后更新 |
| 能力状态 | `ability_state` | 可推断 | 编程、数学、文献阅读、综合应用四个维度的能力评级 | 从知识基础推断 + 答题数据校准 |
| 学习进度 | `progress` | 动态更新 | 当前章节、已完成/进行中知识点、完成率 | 系统根据学习行为自动更新 |

### 2.2 核心原则

**每一维都带三个元数据字段**：`source`（来源）、`confidence`（置信度）、`last_updated`（最近更新时间）。这三个字段是整个画像"可解释性"的基础，比赛演示时每张画像卡片上都要展示"信息来源"和"置信度"。

**弱点和错误分两类**：`self_reported`（学生自己说的）和 `diagnosed`（系统通过答题结果判断的）。一个主观一个客观，来源不同、权重不同、更新方式不同。

**知识基础不用简单的"已掌握/未掌握"二元**：用四级量表（weak / basic / intermediate / advanced），这样路径规划才能区分"需要从头学"和"需要强化巩固"。

**画像标记完整度**：`completeness` 是一个 0-1 的浮点。必问维度各占一定权重，重要维度次之，可推断维度最轻。完整度判断直接决定要不要继续追问。

from pydantic import BaseModel, Field
from datetime import datetime
from typing import Optional

# ==================== 维度 1：专业背景 ====================

class Background(BaseModel):
    """学生专业背景。首轮对话提取，优先级最高。"""
    major: str = ""                        # 专业名称，如"计算机科学与技术"
    grade: int = 1                         # 年级，本科 1-4/研究生 1-3
    school_type: SchoolType = SchoolType.UNDERGRADUATE
    course_foundation: list[str] = Field(default_factory=list)  # 已修课程，如 ["Python", "高等数学"]
    source: str = ""                       # 来源标注，如 "dialogue_round_1"
    confidence: float = 0.0               # 置信度 0-1
    last_updated: Optional[datetime] = None

# ==================== 维度 2：学习目标 ====================

class LearningGoal(BaseModel):
    """学生为什么学这门课。影响路径规划和内容深度。"""
    goal_type: list[GoalType] = Field(default_factory=list)   # 目标类型，可多选
    description: str = ""                   # 学生自述的自然语言原文
    target_course: str = "ml_course"        # 目标课程 ID
    expected_hours_per_week: int = 0        # 每周可投入时间，0 表示未声明
    source: str = ""
    confidence: float = 0.0
    last_updated: Optional[datetime] = None

# ==================== 维度 3：知识基础 ====================

class KnownTopic(BaseModel):
    """一个学生自述的已知主题。用于初始画像，不等同于精确知识点 mastery。"""
    topic: str                             # 主题名称，如 "Python编程"
    level: KnowledgeLevel = KnowledgeLevel.BASIC
    evidence: str = ""                     # 证据简述，如 "学生提到做过课设"

class MasteryEvidence(BaseModel):
    """一次掌握度判断证据。用于解释为什么系统认为某知识点处于某阶段。"""
    source_type: str                       # dialogue / diagnostic_quiz / exercise / code_case / manual
    source_id: str = ""                    # 如 exercise_id、dialogue_round_2
    score_delta: float = 0.0               # 本次证据对 mastery_score 的影响
    confidence_delta: float = 0.0          # 本次证据对判断置信度的影响
    summary: str = ""                      # 前端可展示的证据说明
    created_at: Optional[datetime] = None

class KnowledgePointMastery(BaseModel):
    """按知识图谱节点记录的知识点级掌握状态。路径规划和补强推荐主要读这个结构。"""
    node_id: str                           # 知识图谱 KnowledgePoint uid，如 "ml_gradient_descent"
    node_name: str = ""                    # 展示名称，如 "梯度下降"
    chapter_id: str = ""                   # 所属章节，方便前端分组
    mastery_score: float = 0.0             # 连续掌握分 0-1，后端用于排序和更新
    level: KnowledgeLevel = KnowledgeLevel.WEAK  # 由 mastery_score 映射而来，前端用于简洁展示
    confidence: float = 0.0                # 系统对该判断的置信度 0-1
    attempts: int = 0                      # 相关题目/任务尝试次数
    correct_count: int = 0                 # 答对次数
    last_practiced_at: Optional[datetime] = None
    evidence: list[MasteryEvidence] = Field(default_factory=list)

class KnowledgeBase(BaseModel):
    """学生已有和缺失的知识基础。包含自述主题和知识点级掌握状态。"""
    known_topics: list[KnownTopic] = Field(default_factory=list)
    unknown_topics: list[str] = Field(default_factory=list)  # 明确表示不了解的主题
    node_mastery: dict[str, KnowledgePointMastery] = Field(default_factory=dict)
    source: str = ""
    confidence: float = 0.0
    last_updated: Optional[datetime] = None

# ==================== 维度 4：学习进度（系统自动维护）====================

class Progress(BaseModel):
    """学习进度。不由对话收集，由系统根据学习行为自动更新。"""
    current_chapter_id: Optional[str] = None
    completed_node_ids: list[str] = Field(default_factory=list)
    in_progress_node_ids: list[str] = Field(default_factory=list)
    completion_rate: float = 0.0           # 课程整体完成率
    last_active_at: Optional[datetime] = None
    source: str = "system"                 # 固定为 system
    confidence: float = 1.0               # 系统数据，置信度固定为 1

# ==================== 维度 5：认知风格 ====================

class CognitiveStyleInfo(BaseModel):

    primary: CognitiveStyle = CognitiveStyle.DIAGRAM
    secondary: CognitiveStyle = CognitiveStyle.CODE
    style_ranking: list[CognitiveStyle] = Field(default_factory=list)  # 完整偏好排序
    source: str = ""
    confidence: float = 0.0

class SelfReportedWeakPoint(BaseModel):

### 2.4 知识点级掌握状态

`knowledge_base.known_topics` 只表示学生自述的基础，例如“学过 Python”“高数还行”。它适合做初始画像，但不能直接判断某个机器学习知识点是否学到位。真正用于 GraphRAG、路径规划、练习推荐和补强判断的，应当是 `knowledge_base.node_mastery`。

核心原则：

- **每个知识点单独建账**：以知识图谱 `KnowledgePoint.uid` 为 key，记录该知识点的 `mastery_score`、`level`、`confidence`、答题次数和证据。
- **等级只是展示标签**：`weak / basic / intermediate / advanced` 用于前端简洁展示；后端排序和更新使用连续的 `mastery_score`。
- **证据必须可追溯**：每次自述、诊断题、练习题、代码任务都写入 `evidence`，解释为什么系统认为学生掌握或薄弱。
- **能力水平由知识点聚合而来**：数学能力、编程能力、应用能力不是手填标签，而是由相关知识点和题型表现聚合得到。

推荐分数映射：

| `mastery_score` | 展示等级 | 判断含义 | 推荐动作 |
|---|---|---|---|
| `0.00 - 0.25` | `weak` | 概念不清，容易混淆，前置知识可能缺失 | 先补概念和前置知识 |
| `0.25 - 0.55` | `basic` | 知道概念，但计算、代码或迁移应用不稳定 | 给图解、例题和低中难度练习 |
| `0.55 - 0.80` | `intermediate` | 能完成常规题，能解释主要过程 | 给综合题和轻量项目任务 |
| `0.80 - 1.00` | `advanced` | 能迁移到新场景，能解释误区和边界条件 | 给挑战题、案例分析或扩展知识 |

注意：`advanced` 不能只靠一道选择题判定。至少需要更强证据，例如中高难度题连续答对、代码题通过、案例分析答得完整，或者在多个相关知识点上表现稳定。

知识点 mastery 的更新可以先用 MVP 规则：

```python
def update_node_mastery(
    current_score: float,
    is_correct: bool,
    difficulty: int,
    cognitive_level: str,
    used_hint: bool = False,
) -> float:
    """根据一次练习结果更新某个知识点的掌握分。"""
    difficulty_weight = 0.02 + difficulty * 0.01
    task_weight = {
        "remember": 0.6,
        "understand": 0.8,
        "apply": 1.0,
        "analyze": 1.2,
        "create": 1.3,
    }.get(cognitive_level, 1.0)
    hint_penalty = 0.5 if used_hint else 1.0

    if is_correct:
        delta = difficulty_weight * task_weight * hint_penalty
    else:
        delta = -difficulty_weight * 1.2

    return max(0.0, min(1.0, current_score + delta))
```

能力状态的聚合示例：

```python
ABILITY_NODE_GROUPS = {
    "mathematics": [
        "ml_linear_algebra_basic",
        "ml_probability_statistics_basic",
        "ml_calculus_optimization_basic",
        "ml_gradient_descent",
    ],
    "programming": [
        "ml_python_basic",
        "ml_sklearn_basic",
        "ml_gradient_descent_code",
    ],
    "application": [
        "ml_model_evaluation",
        "ml_overfitting_regularization",
        "ml_kmeans",
    ],
}

def aggregate_ability_level(profile: StudentProfile, ability_key: str) -> AbilityLevel:
    """由相关知识点 mastery 聚合能力水平。"""
    node_ids = ABILITY_NODE_GROUPS[ability_key]
    scores = [
        profile.knowledge_base.node_mastery[node_id].mastery_score
        for node_id in node_ids
        if node_id in profile.knowledge_base.node_mastery
    ]
    if not scores:
        return AbilityLevel.BASIC

    avg_score = sum(scores) / len(scores)
    if avg_score < 0.25:
        return AbilityLevel.WEAK
    if avg_score < 0.55:
        return AbilityLevel.BASIC
    if avg_score < 0.80:
        return AbilityLevel.INTERMEDIATE
    return AbilityLevel.ADVANCED
```

这样前端可以展示“整体数学能力 basic”，但后端仍然能指出具体原因：比如“梯度下降 weak、微积分优化 basic、线性代数 intermediate”，从而精确推荐补强路径。

### 2.5 完整度计算规则

```python
def compute_completeness(profile: StudentProfile) -> float:
    """加权计算画像完整度。每个维度有预设权重。"""
    weights = {
        "background":      0.20,   # 必问
        "learning_goal":   0.20,   # 必问
        "knowledge_base":  0.20,   # 必问
        "cognitive_style": 0.10,   # 重要
        "preferences":     0.10,   # 重要
        "weak_points":     0.08,   # 可推断（自述部分算分，诊断部分不算）
        "ability_state":   0.07,   # 可推断
        "progress":        0.05,   # 动态更新，初始 0 分是正常的
    }

    dim_scores = {
        "background":      1.0 if profile.background.major and profile.background.grade > 0 else 0.0,
        "learning_goal":   1.0 if profile.learning_goal.goal_type or profile.learning_goal.description else 0.0,
        "knowledge_base":  0.5 if profile.knowledge_base.known_topics else 0.3 if profile.knowledge_base.unknown_topics else 0.0,
        "cognitive_style": 1.0 if profile.cognitive_style.primary else 0.0,
        "preferences":     1.0 if profile.preferences.resource_ranking else 0.0,
        "weak_points":     1.0 if profile.weak_points.self_reported else 0.0,
        "ability_state":   0.5 if profile.ability_state.programming != AbilityLevel.BASIC or
                                  profile.ability_state.mathematics != AbilityLevel.BASIC else 0.0,
        "progress":        profile.progress.completion_rate,
    }

    return sum(weights[k] * dim_scores[k] for k in weights)
```

### 2.6 数据库持久化设计

学生画像是核心业务数据，不能只用内存字典临时保存。画像需要支持跨会话恢复、前端展示、更新历史追踪、答题后动态更新、后续统计分析，所以实现时应直接落数据库。

推荐采用“双数据库职责分离”：

| 数据类型 | 存储位置 | 原因 |
|---|---|---|
| 课程知识图谱、知识点、章节、资源关系 | Neo4j | 图遍历、前置关系、GraphRAG 扩展更自然 |
| 学生画像、画像会话、答题记录、画像更新历史 | PostgreSQL / MySQL / SQLite | 业务数据强一致、事务更新、时间线追踪更方便 |
| 知识点掌握度 `node_mastery` | 关系型数据库单独表 | 需要按学生、知识点、章节、分数、更新时间高频查询 |

如果本地演示环境不想额外启动 PostgreSQL，可以用 SQLite 跑同一套 SQLAlchemy/SQLModel 模型；但代码结构必须按关系型数据库设计，不再使用内存 store。比赛部署或团队协作时直接把 `DATABASE_URL` 切到 PostgreSQL。

#### 事务边界

一次画像更新必须放在同一个事务里：

1. 更新 `student_profiles` 主表。
2. upsert 相关 `student_node_mastery`。
3. 插入 `mastery_evidence`。
4. 插入 `profile_update_events`。

这样前端不会看到“画像卡片更新了，但时间线没记录”或“答题记录写了，但 mastery 没变”的半完成状态。

### 2.7 画像初始化与画像更新的边界

画像初始化和画像更新必须分开设计。两者可以复用同一套 LLM 抽取、知识点对齐、画像合并和数据库事务能力，但入口、触发来源、可信度、写入字段和前端展示语义都不同。

| 对比项 | 画像初始化 | 画像更新 |
|---|---|---|
| 核心问题 | 第一次认识这个学生是谁 | 学生学习后发生了什么变化 |
| 主要目标 | 快速建立可用画像，让系统开始个性化推荐 | 根据新证据持续校准画像、掌握度和路径 |
| 典型触发 | 首次自我介绍、导入资料、预设 demo profile | 答题结果、代码练习、学习进度、后续对话、手动修正 |
| 主要证据 | 学生自述，置信度中低 | 学习行为和测评结果，置信度更高 |
| 主要写入 | `background`、`learning_goal`、`known_topics`、`preferences`、`cognitive_style`、`weak_points.self_reported` | `student_node_mastery`、`weak_points.diagnosed`、`progress`、`mastery_evidence`、`profile_update_events` |
| LLM 角色 | 从首次对话中抽取初始画像 | 从后续对话中抽取变化；答题/进度更新通常不需要 LLM |
| 前端语义 | “完成建档”“还缺哪些信息” | “画像发生了什么变化”“为什么推荐改变了” |

因此服务层建议拆成三类能力：

```text
ProfileInitializationService
    init_from_dialogue()
    init_from_imported_profile()
    init_demo_profile()

ProfileUpdateService
    update_from_dialogue()
    update_from_exercise_result()
    update_from_learning_progress()
    manual_patch()
    system_recompute()

ProfileMergeService
    merge_profile_delta()
    apply_mastery_update()
    write_update_event()
    write_mastery_evidence()
```

底层共用能力：

```text
ProfileExtractor      # LLM/规则抽取画像增量
ProfileNormalizer     # 知识点名称 -> KnowledgePoint uid，对齐图谱
ProfileRepository     # 关系型数据库 CRUD / upsert / transaction
CompletenessService   # 完整度和缺失维度判断
DashboardBuilder      # 完整画像 -> 前端 dashboard DTO
```

事件类型也要显式区分：

| trigger | 触发场景 | 说明 |
|---|---|---|
| `init_dialogue` | 首次对话建档 | 更新基础画像，完整度通常明显上升 |
| `update_dialogue` | 后续画像补充对话 | 只记录新增或修正的信息 |
| `exercise_result` | 练习/测评提交 | 更新 `node_mastery`、诊断弱点和证据 |
| `learning_progress` | 学习节点完成、资源浏览 | 更新学习进度 |
| `manual_patch` | 学生手动修正画像 | 优先级高，前端应显示为用户确认 |
| `system_recompute` | 系统重算能力状态或弱点排序 | 由后台聚合规则触发 |

初始化阶段可以根据自述创建低置信度 `node_mastery`，但不能直接给 `advanced`。后续只有答题、代码任务、案例分析等行为证据，才应显著提高或降低某知识点掌握分。

## 三、画像初始化对话状态机设计

画像初始化不是无状态的"用户说一句、LLM 抽一下"，而是一个**有状态的首次建档流程**。每轮对话要能判断：当前在第几轮、哪些维度已经填了、哪些还需要问、该问什么。后续学习过程中的画像更新不走这个状态机，而是由 `ProfileUpdateService` 根据事件类型触发。

### 3.1 对话状态

```python
class ProfileBuildSession(BaseModel):
    """一次画像构建会话的状态"""
    session_id: str                       # 会话 ID
    student_id: str                       # 关联的学生
    current_round: int = 0               # 当前轮次（从 0 开始，0 = 系统开场白）
    max_rounds: int = 6                  # 最大对话轮数
    status: str = "building"             # building / completed / timeout / cancelled

    profile_snapshot: StudentProfile     # 当前轮画像快照
    filled_dimensions: list[str] = Field(default_factory=list)   # 已填充的维度名
    missing_required: list[str] = Field(default_factory=list)    # 缺失的必问维度
    missing_important: list[str] = Field(default_factory=list)   # 缺失的重要维度

    conversation_history: list[dict] = Field(default_factory=list)  # [{role, content, extracted_fields}, ...]
```

### 3.2 对话状态流转

```text
         ┌──────────┐
         │  START   │
         └────┬─────┘
              │ 系统发开场白（"简单说说你的情况"）
              ▼
      ┌───────────────┐
      │  BUILDING     │◄──── 每轮对话循环
      │  round: 1..N  │
      └───────┬───────┘
              │ 用户回复
              ▼
      ┌───────────────┐
      │ ① LLM 抽取   │  从本轮对话中提取结构化字段
      │   增量字段    │  只返回本轮真正说了的内容
      └───────┬───────┘
              │
              ▼
      ┌───────────────┐
      │ ② 合并画像   │  将增量字段合并到 profile_snapshot
      │   重算完整度  │  更新 completeness
      │   记录更新日志│  写入 update_history
      └───────┬───────┘
              │
              ▼
      ┌───────────────┐
      │ ③ 缺失检测   │  检查 missing_required / missing_important
      └───┬───────┬───┘
          │       │
    有缺失 │       │ 无缺失 或 达到最大轮数
          │       │
          ▼       ▼
    ┌─────────┐  ┌──────────────┐
    │④ 生成   │  │  COMPLETED   │
    │  追问   │  │  画像完成    │
    └────┬────┘  └──────────────┘
         │
         ▼
    返回系统回应 + 画像快照 → 进入下一轮
```

### 3.3 追问生成策略

追问不是随机选一个缺失维度问，而是按优先级 + 自然度生成。

**追问顺序规则**：

```python
QUESTION_TEMPLATES = {
    "background": [
        "方便说一下你现在学什么专业、大几了吗？",
        "你之前上过哪些跟编程或数学相关的课？",
    ],
    "learning_goal": [
        "你学机器学习主要是为了什么？比如准备考试、做课程项目，还是为竞赛或找工作做准备？",
        "你大概每周能花多少时间在这门课上？",
    ],
    "knowledge_base": [
        "你觉得 Python 编程和数学基础分别是什么水平？有没有哪块感觉特别吃力？",
        "在机器学习相关的数学（线性代数、概率统计、微积分）里，你有没有觉得特别薄弱的部分？",
    ],
    "cognitive_style": [
        "你更喜欢哪种学习方式？看图解和动画、读代码案例、做练习题，还是看文字讲义？",
    ],
    "preferences": [
        "你喜欢的资源类型排个序吧：文字讲义、图解导图、练习题、视频脚本、代码案例。",
        "你一次一般能集中学多久？喜欢从基础慢慢来，还是直接挑战有难度的内容？",
    ],
}
```

**追问选择算法**：

```python
def select_next_question(session: ProfileBuildSession) -> str | None:
    """选一条最合适的追问"""
    # 优先问必问维度
    if session.missing_required:
        dim = session.missing_required[0]  # 按 priority 排好的
        return QUESTION_TEMPLATES[dim][0]

    # 再问重要维度
    if session.missing_important:
        dim = session.missing_important[0]
        return QUESTION_TEMPLATES[dim][0]

    # 都没有缺失，完成
    return None
```

### 3.4 何时结束

三种结束条件，任一触发即进入 COMPLETED：

1. **完整度达标**：`completeness >= 0.75`（必问 + 重要维度都已覆盖），且无缺失必问维度
2. **达到最大轮数**：`current_round >= max_rounds`，即使还有缺失也结束，不强行追问
3. **用户主动结束**：用户说"可以了""先这样""开始学习吧"等

## 四、LLM 抽取层设计

这是画像模块最核心的 AI 能力。每轮对话需要 LLM 从对话文本中提取结构化字段。但要注意：初始化抽取和更新抽取不是同一个语义。

### 4.1 不是"LLM 直接填 8 维"

注意关键设计决策：**LLM 每次只提取本轮对话中真正出现的信息，不脑补、不推断。** 画像的完整性是通过多轮对话逐步累积的，不是一次 LLM 调用填满的。

### 4.1.1 初始化抽取与更新抽取

LLM 抽取层建议支持两个模式：

| 模式 | 使用入口 | 抽取重点 | 禁止行为 |
|---|---|---|---|
| `initialization` | `POST /api/profile/init` | 尽量覆盖首次建档所需的基础维度，如背景、目标、已学课程、偏好、自述薄弱点 | 不要生成诊断弱点，不要给高置信 mastery |
| `update` | `POST /api/profile/chat` 或后续补充对话 | 只抽取相对于已有画像新增、修正或冲突的信息 | 不要重复输出已有字段，不要重置完整画像 |

初始化 prompt 可以问“请从首次自我介绍中抽取初始画像信息”；更新 prompt 必须更严格：“请只抽取本轮相对于已有画像发生变化的信息”。

答题结果、代码运行结果、学习进度更新通常不需要 LLM 抽取，应由 `ProfileUpdateService` 根据结构化事件直接更新 `student_node_mastery`、`progress` 和 `mastery_evidence`。

### 4.2 ProfileExtractor 接口

```python
class ProfileExtractor:
    """负责从单轮对话中提取增量画像字段"""

    async def extract(
        self,
        user_message: str,
        system_reply: str,
        existing_profile: StudentProfile,
        round_number: int,
        mode: str = "update",  # initialization / update
    ) -> ProfileExtractionResult:
        """
        输入：
          - user_message: 学生本轮说的内容
          - system_reply: 系统本轮的回话（如果有追问，包含追问内容）
          - existing_profile: 当前已累积的画像
          - round_number: 第几轮对话
          - mode: 初始化抽取还是更新抽取
        输出：
          - 本轮新提取到的字段（增量，不是全量）
          - 每个字段带 confidence
          - 没有提取到的字段不出现（不是空值）
        """
```

### 4.3 抽取返回结构

```python
class ProfileExtractionResult(BaseModel):
    """单轮对话的抽取结果——只包含本轮真正提到了的信息"""
    round_number: int
    source_label: str                     # 如 "dialogue_round_2"，用于标注 source

    background_update: Optional[Background] = None      # 本轮更新的 background
    learning_goal_update: Optional[LearningGoal] = None
    knowledge_base_update: Optional[KnowledgeBase] = None
    cognitive_style_update: Optional[CognitiveStyleInfo] = None
    weak_points_update: Optional[WeakPoints] = None
    preferences_update: Optional[Preferences] = None
    ability_state_update: Optional[AbilityState] = None

    confidence_notes: list[str] = Field(default_factory=list)  # 置信度低的说明
```

**原则**：`Optional[...] = None` 表示本轮对话中没提到这个维度，不要为了"填满"而编造内容。

### 4.4 LLM Prompt 设计

```
你是 EduGraph-Agent 的 Profile Agent。
你的任务是**仅从本轮对话中**提取学生明确表达的个人信息，填入对应维度。
不要推断学生没有明确说过的内容。

## 8 个维度及其字段

1. background: 专业(major)、年级(grade)、学校类型(school_type)、已修课程(course_foundation)
2. learning_goal: 目标类型(goal_type)、自然语言描述(description)、每周投入时间(expected_hours_per_week)
3. knowledge_base: 已知主题列表(known_topics: [{topic, level}])、明确不知道的主题(unknown_topics)
   level 可选值: weak / basic / intermediate / advanced
4. cognitive_style: 主要风格(primary)、次要风格(secondary)、完整排序(style_ranking)
   风格可选值: theory(偏理论) / case(偏案例) / code(偏代码) / diagram(偏图解)
5. weak_points: 自述薄弱点(self_reported: [{topic, description}])
   不要编造 node_id；如果提到薄弱点，只输出自然语言知识点名，后端会负责图谱对齐
6. preferences: 资源偏好排序(resource_ranking)、单次学习时长(session_length)、难度偏好(difficulty_preference)
   资源类型: document(讲义) / diagram(图解) / exercise(练习题) / video_script(视频脚本) / code_case(代码)
   学习时长: short(≤20min) / medium(20-45min) / long(>45min)
   难度偏好: gradual(循序渐进) / challenge(挑战) / review(复习)
7. ability_state: 编程(programming)、数学(mathematics)、文献阅读(reading_papers)、应用(application)
   每个能力: weak / basic / intermediate / advanced
8. progress: **对话中不提取**，由系统自动维护

## 重要规则

1. 只提取学生在本轮对话中**明确表达**的信息
2. 如果学生说"我觉得我编程还行"，可以提 ability_state.programming = intermediate，confidence 设为 0.7
3. 如果学生说"高数考了 90"，可以提 knowledge_base.known_topics 中加入 {topic: "高等数学", level: "advanced"}
4. 如果本轮完全没有涉及某个维度，该维度返回 null，不要填默认值
5. 如果 mode=update，只输出新增、修正或冲突的信息，不要重复已有画像
6. progress、diagnosed weak_points、exercise-based mastery 不由 LLM 提取
7. confidence 表示你对"信息准确反映学生真实情况"的把握：
   - 0.9+: 学生明确、具体地描述了该信息
   - 0.7-0.89: 有明确提及但细节不够
   - 0.5-0.69: 仅从表达中间接推断
   - <0.5: 高度不确定，宁可本轮不提

## 当前已累积的画像

{existing_profile_json}

## 本轮对话（第 {round_number} 轮）

系统：{system_reply}
学生：{user_message}

请输出本轮的增量提取结果。只填本轮涉及的维度。
```

### 4.5 合并逻辑

```python
def merge_extraction_to_profile(
    profile: StudentProfile,
    extraction: ProfileExtractionResult,
    round_number: int,
) -> StudentProfile:
    """将增量抽取结果合并到画像，生成更新记录"""
    source_label = f"dialogue_round_{round_number}"
    updated_fields = []

    # 每个维度：如果本轮有更新，覆盖对应字段
    if extraction.background_update is not None:
        extraction.background_update.source = source_label
        profile.background = extraction.background_update
        updated_fields.append("background")

    if extraction.learning_goal_update is not None:
        extraction.learning_goal_update.source = source_label
        profile.learning_goal = extraction.learning_goal_update
        updated_fields.append("learning_goal")

    # ... 其他维度同理

    # 如果有更新，写 update_history
    if updated_fields:
        profile.update_history.append(ProfileUpdateRecord(
            trigger="init_dialogue" if round_number == 1 else "update_dialogue",
            trigger_detail=source_label,
            updated_fields=updated_fields,
            summary=f"第 {round_number} 轮对话更新了: {', '.join(updated_fields)}",
        ))

    # 重算完整度
    profile.completeness = compute_completeness(profile)
    profile.updated_at = datetime.utcnow()

    return profile
```

## 五、画像如何被已有系统消费

画像不是独立模块，它要深度嵌入现有的 GraphRAG、诊断和资源生成链路。

### 5.1 画像 → GraphRAG 个性化检索

当前 `EvidenceRetriever` 已经消费了 `weak_points` 做练习排序。需要扩展到消费更多维度：

| 画像维度 | GraphRAG 消费方式 | 实现位置 |
|---|---|---|
| `weak_points` | 练习题排序加权（已有） + 前置知识扩展时优先补齐薄弱点的前置节点 | `ranking.py` + `GraphExpansionPolicy` |
| `cognitive_style` | `diagram` → 多取 SUPPORTS chunk；`code` → 多取 PRACTICES code_cases | `GraphExpansionPolicy` |
| `preferences.resource_ranking` | 调整 Evidence Package 中各类证据的数量分配 | `EvidenceRetriever` |
| `ability_state` | 过滤难度超出能力 +1 的节点 | `GraphExpansionPolicy.difficulty_filter()` |
| `knowledge_base` | 跳过学生 already known 的节点 | `GraphExpansionPolicy.skip_known()` |
| `learning_goal` | 优先推荐与目标类型匹配的知识点和资源 | `EvidenceRetriever` ranking |

### 5.1.1 兼容适配策略

短期实现上保留两层模型：

- `StudentProfile`：完整画像模型，用于画像构建、画像页面展示、画像更新和长期存储。
- `StudentProfileInput`：轻量查询画像，用于 GraphRAG、诊断推荐和资源生成请求中的个性化参数。

在 `StudentProfile` 上提供转换方法即可：

```python
def to_graphrag_input(profile: StudentProfile) -> StudentProfileInput:
    """把完整画像压缩成当前 GraphRAG/Agent 可消费的轻量输入。"""
    return StudentProfileInput(
        weak_points=[
            item.topic for item in profile.weak_points.self_reported
        ] + [
            item.node_id for item in profile.weak_points.diagnosed
        ],
        preferences=[item.value for item in profile.preferences.resource_ranking],
        goal=profile.learning_goal.description,
        mastery={
            topic.topic: topic.level.value
            for topic in profile.knowledge_base.known_topics
        },
    )
```

等前端和画像 API 稳定后，再让 GraphRAG 和 Agent 直接消费完整 `StudentProfile`。这样迭代风险更低，也符合当前项目“先跑通闭环，再扩大模型”的节奏。

### 5.2 画像 → 资源生成个性化

当前 `ResourceGenerationService` 已经接收 profile 参数并注入 LLM prompt。需要把完整 8 维传进去：

```python
# resource_agents.py 的 system prompt 中加入
"学生画像："
f"- 偏好学习风格：{profile.cognitive_style.primary} + {profile.cognitive_style.secondary}"
f"- 资源类型偏好：{profile.preferences.resource_ranking}"
f"- 薄弱知识点：{[w.topic for w in profile.weak_points.self_reported]}"
f"- 数学水平：{profile.ability_state.mathematics}"
f"- 编程水平：{profile.ability_state.programming}"
```

这样 Document Agent 会生成更适合图解倾向学生的讲解，Code Agent 会生成匹配学生编程水平的代码复杂度。

### 5.3 画像更新 ← 练习评估

这是闭环最关键的一环。当前 `Exercise` 的 `profile_update_policy` 已经定义好了：

```json
{
  "on_wrong": {
    "mastery_delta": -0.05,
    "add_weak_points": ["ml_gradient_descent"],
    "recommend_review_nodes": ["ml_loss_function"]
  }
}
```

需要新增 `ProfileUpdateService` 来执行这个策略：

```python
class ProfileUpdateService:
    async def apply_exercise_result(
        self,
        student_id: str,
        exercise_id: str,
        is_correct: bool,
    ) -> StudentProfile:
        """答题后更新画像"""
```

更新逻辑：
1. 答错 → `weak_points.diagnosed` 中对应节点的 `error_rate` 增加、`total_attempts` 增加
2. 答对 → `weak_points.diagnosed` 中对应节点的 `error_rate` 略微下降
3. 无论对错 → 更新 `progress.in_progress_node_ids`
4. 写出 `update_history` 记录

## 九、不建议现在做的事

1. **不要把画像主数据塞进 Neo4j** — Neo4j 继续负责课程知识图谱；画像、mastery、更新历史进关系型业务库。这样既保留图谱检索优势，也方便画像事务更新、时间线查询和前端展示

## 十一、前端展示与交互设计

学生画像不只是后端数据结构，也是比赛演示中最容易让评委理解“个性化”的界面。前端目标不是把 JSON 全量铺出来，而是把画像转成一个清晰、可信、可更新的学习者仪表盘。

### 11.1 页面信息架构

推荐把画像页做成一个工作台式布局，而不是表单页或聊天页。

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部摘要：学生身份 / 学习目标 / 完整度 / 最近更新             │
├───────────────┬──────────────────────────────┬─────────────┤
│ 画像维度卡片区 │ 知识掌握与薄弱点区              │ 更新历史     │
│ 8 张紧凑卡片   │ 进度条、掌握矩阵、弱点标签        │ 时间线       │
├───────────────┴──────────────────────────────┴─────────────┤
│ 个性化解释区：为什么推荐这些前置知识、资源和下一步路径         │
└────────────────────────────────────────────────────────────┘
```

首屏必须让用户一眼看到 4 件事：

1. 这个学生是谁：专业、年级、目标课程。
2. 系统了解到了多少：画像完整度、缺失维度。
3. 个性化依据是什么：学习目标、偏好、薄弱点、能力状态。
4. 刚刚发生了什么：本轮对话或答题更新了哪些字段。

### 11.2 视觉组件设计

| 区域 | 展示内容 | 组件建议 | 设计要点 |
|---|---|---|---|
| 顶部摘要 | 专业年级、目标、完整度、更新时间 | 横向 summary bar + 环形进度 | 信息密度高，避免大面积装饰 |
| 8 维画像卡片 | background / goal / knowledge / progress / style / weak_points / preferences / ability | 2-4 列响应式卡片 | 每张卡片半屏内可读，圆角不超过 8px |
| 置信度 | 每个维度的 `confidence` | 小型进度条或 badge | 低于 0.6 用提示色，允许用户修正 |
| 信息来源 | `source`、`last_updated` | tooltip / secondary text | 不抢主视觉，但悬停可解释 |
| 知识掌握 | 已知主题、未知主题、mastery | 水平条 / 小型热力矩阵 | 使用知识点名称 + 等级，不展示原始枚举 |
| 薄弱点 | 自述弱点、诊断弱点 | 标签组 + 排序列表 | 区分“自述”和“系统诊断” |
| 学习偏好 | 资源偏好、时长、难度 | 图标标签 / segmented badges | 用图标强化 document、diagram、code_case 等类型 |
| 更新历史 | `update_history` | 时间线 | 体现画像会随学习动态变化 |
| 个性化解释 | GraphRAG/资源生成为什么这样推荐 | explain panel | 连接画像与推荐结果，增强可信度 |

### 11.3 8 维卡片展示规则

每张卡片采用统一结构，前端便于复用：

```text
┌────────────────────┐
│ 图标 + 维度名称       │
│ 核心结论 1-2 行       │
│ 关键标签 / 进度条      │
│ 来源 · 置信度 · 编辑   │
└────────────────────┘
```

建议的卡片摘要：

- 专业背景：`计算机科学与技术 · 大二 · 已学 Python/高数`
- 学习目标：`课程项目 · 每周 4 小时 · 目标：理解神经网络训练`
- 知识基础：`Python intermediate，梯度下降 weak`
- 学习进度：`当前 ch06，完成 18%`
- 认知风格：`图解优先，其次代码`
- 易错点：`梯度下降（自述），反向传播链式法则（诊断）`
- 学习偏好：`diagram > code_case > document，循序渐进`
- 能力状态：`编程 intermediate，数学 basic，应用 basic`

卡片不要展示大段原始 JSON。需要细节时点击卡片进入抽屉面板，展示字段详情、来源、置信度、更新时间和手动修正入口。

### 11.4 前端友好的 DTO

后端可以保留完整 `StudentProfile`，但前端画像页建议额外提供一个展示 DTO，减少前端重复拼装逻辑：

```python
class ProfileDimensionView(BaseModel):
    key: str
    title: str
    icon: str
    summary: str
    tags: list[str] = Field(default_factory=list)
    confidence: float = 0.0
    source: str = ""
    last_updated: datetime | None = None
    status: str = "filled"  # filled / missing / inferred / low_confidence
    editable: bool = True

class ProfileDashboardResponse(BaseModel):
    student_id: str
    display_name: str
    headline: str
    completeness: float
    missing_dimensions: list[str]
    dimensions: list[ProfileDimensionView]
    weak_point_rank: list[dict]
    mastery_overview: list[dict]
    recent_updates: list[ProfileUpdateRecord]
    personalization_reasons: list[str]
```

建议新增接口：

```text
GET /api/profile/{student_id}/dashboard
```

这个接口专门服务前端画像页。`GET /api/profile/{student_id}` 继续返回完整模型，供调试和后续高级页面使用。

### 11.5 交互状态

画像页至少需要覆盖这些状态：

1. **构建中**：对话刚开始，顶部显示完整度，缺失维度卡片置灰，并给出下一条追问。
2. **已更新**：每轮对话或答题后，高亮刚更新的卡片 1-2 秒，并在时间线新增记录。
3. **低置信度**：卡片显示“待确认”，点击后可以让学生手动修正。
4. **诊断变化**：答错练习后，薄弱点卡片和知识掌握区同步变化。
5. **推荐解释**：当 GraphRAG 或资源生成返回结果时，展示“因为你偏好图解且梯度下降较薄弱，所以优先推荐……”

这些状态能让前端看起来不是静态档案，而是一个会随学习行为变化的智能画像。

### 11.7 审美与可读性约束

- 页面应像学习分析仪表盘，不要做成营销页或大 hero 页。
- 卡片只用于独立信息单元，不做卡片套卡片。
- 单卡片控制在 3-5 行核心信息，长内容进入抽屉或详情弹窗。
- 不用单一蓝紫渐变铺满页面；建议用中性背景 + 少量语义色区分“已掌握 / 薄弱 / 待确认 / 推荐”。
- 图标优先使用现成图标库，比如 `User`、`Target`、`BookOpen`、`ChartNoAxesColumnIncreasing`、`Brain`、`AlertTriangle`、`SlidersHorizontal`、`Activity`。
- 移动端改为单列卡片 + 折叠时间线；桌面端使用 12 栅格布局。

### 11.8 演示脚本中的画像镜头

比赛演示中建议给画像页 40-60 秒：

1. 学生输入：“我是计算机大二，学过 Python 和高数，想理解神经网络训练，更喜欢图解和代码。”
2. 页面实时生成 8 维画像卡片，完整度从 0% 增长到约 75%。
3. 点击“开始学习”，GraphRAG 查询反向传播，解释区显示：因为学生“梯度下降薄弱 + 偏好图解/代码”，所以优先补前置知识并生成图解/代码资源。
4. 学生做错一道梯度下降题，薄弱点卡片新增“梯度下降（系统诊断）”，更新时间线出现记录。
5. 路径推荐刷新，展示“先复习梯度下降，再学习反向传播”。

这个镜头能把“画像构建 -> 个性化检索 -> 资源生成 -> 学习评估 -> 画像更新”的闭环讲清楚，比单独展示接口返回 JSON 更有说服力。
