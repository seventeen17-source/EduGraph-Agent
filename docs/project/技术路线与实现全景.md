# EduGraph-Agent 技术路线与实现全景

> 本文回答四个问题：项目用了哪些 **Skill / 工程约束**、用了哪些 **技术**、**技术路线**是什么、以及**每一步靠什么实现**。
> 定位：基于「知识图谱 + LLM」的个性化学习平台，覆盖学习助手、资源生成、练习评估、学生画像、成长追踪五大能力。

---

## 一、项目用到的「Skill / 工程方法」

这里的 Skill 有两层含义：一是给 AI 编码助手（Claude Code / Codex / Trae）用的**技能定义**，二是团队采用的**工程方法论**。

### 1.1 AI 助手技能（Skill 文件）

项目为 AI 编码工具维护了统一的技能定义，保证 AI 在改代码前就有一致的项目上下文：

| 位置 | 面向工具 | 作用 |
|------|---------|------|
| `.trae/skills/edugraph/SKILL.md` | Trae（主力工具，随仓库提交） | 完整的项目开发技能（技术栈/目录/路由/约定/工作流） |

> 开发期间也曾为 Claude Code / Codex 维护过等价副本（`.claude/` / `.codex/`），仓库中仅保留 Trae 版本作为唯一权威来源，避免多份拷贝漂移。

`SKILL.md` 本质是一份「项目开发说明书」，让 AI 在改代码前就知道：技术栈是什么、目录怎么摆、路由怎么注册、有哪些**必须遵守的约定**（见第五节），从而减少来回试错。

### 1.2 Spec-Driven Development（规格驱动开发）

项目采用 **Kiro / Trae 风格的 spec 工作流**，每个大改动先写规格再动手：

```
.trae/specs/<feature>/
├── spec.md       # Why / What Changes / Impact / ADDED·MODIFIED·REMOVED Requirements + Scenario
├── tasks.md      # 拆解到可执行的任务清单（P0/P1/P2 分级）
└── checklist.md  # 验收/演示检查项
```

当前两个活跃 spec：
- `refactor-exercise-assessment` — 练习与评估系统重构
- `enhance-system-intelligence` — 系统智能化增强（让每个决策「可解释、可追溯、可自修复」）

spec.md 的写法是 **EARS 风格需求 + Scenario（WHEN/THEN/AND）**，每条需求都能对应到验收场景，也直接标注了受影响的后端/前端文件，便于闭环落地。

## 二、技术栈总览

### 后端（Python / FastAPI）

| 技术 | 用途 | 靠它实现的能力 |
|------|------|--------------|
| FastAPI + Uvicorn | Web 框架 / ASGI | 38 个 REST 端点、lifespan 生命周期管理 |
| SQLAlchemy 2.0 + aiosqlite | 异步 ORM / SQLite | 会话、练习记录、画像、资源记录持久化 |
| Neo4j 5.x | 知识图谱存储 | 知识点节点、前置依赖边、子图查询 |
| ChromaDB | 向量库 | 语义记忆 + 课程语义视图检索 |
| LangChain + LangGraph | LLM 编排 | 对话图、Hybrid RAG 工作流、资源生成 Agent |
| OpenAI SDK | LLM 调用 | 生成/评分/抽取（默认 gpt-4o-mini，可配 base_url） |
| python-jose + bcrypt | 认证 | JWT 签发/校验、密码哈希 |
| sse-starlette | 流式 | 助手对话 SSE 推送 |

### 前端（Vue 3 / TypeScript）

| 技术 | 用途 |
|------|------|
| Vue 3 + Vite 6 + TS | 核心框架 + 构建 |
| Pinia | 状态管理（auth / profile / assistant / learning 四个 store） |
| Vue Router 4 | 16 条路由 + 登录守卫 + 401 自动刷新 |
| Element Plus | UI 组件库 |
| ECharts | 雷达图、热力图等画像可视化 |
| markmap-lib / Mermaid / KaTeX / highlight.js / markdown-it | 思维导图 / 流程图 / 公式 / 代码高亮 / Markdown 渲染 |

## 三、技术路线（系统架构分层）

```
┌────────────────────────────────────────────────────────┐
│  前端 Vue 3：15 个页面 · 4 个 Pinia store · Axios 封装      │
└───────────────────────────┬────────────────────────────┘
                            │ REST + SSE
┌───────────────────────────┴────────────────────────────┐
│  API 层 FastAPI：9 个路由模块 · deps.py 依赖注入            │
├──────────────────────────────────────────────────────────┤
│  服务层（ServiceCache 单例管理）                            │
│  Assistant · ResourceGeneration · GraphRAG · Diagnosis   │
│  · Profile · ExerciseRecord                              │
├──────────────────────────────────────────────────────────┤
│  编排层 LangChain / LangGraph                             │
│  对话图 · Hybrid RAG 图 · 5 个资源生成 Agent（含自修复）     │
├──────────────────────────────────────────────────────────┤
│  存储层                                                    │
│  Neo4j(图谱)  ChromaDB(向量)  SQLite(关系: 画像/练习/会话)  │
└──────────────────────────────────────────────────────────┘
```

### 核心学习闭环

```
用户提问 → 学习助手(AssistantService) → GraphRAG(HybridRAG) → 返回证据
   ↓
资源生成(5 个 Agent 并行生成，生成-校验-修复) → 存入 GeneratedResourceRecord
   ↓
练习与评估 → 按题型评分(choice/short_answer/coding/case_analysis)
   ↓
画像更新(ProfileService) → NodeMastery 增量更新 → 证据链记录来源
   ↓
学习成长 → 时间轴 + 遗忘预警 → 触发复习推荐（回到闭环起点）
```

## 四、每一步依靠什么实现

| 步骤 | 依赖实现 | 关键文件 |
|------|---------|---------|
| **1. 认证** | JWT（python-jose）+ bcrypt，前端 Axios 拦截器自动注入 token、401 自动刷新 | `app/auth/jwt.py`、`frontend/src/api/client.ts` |
| **2. 学习助手对话** | LangGraph 对话图 + 工具调用（资源生成/练习/记忆），SSE 流式返回 | `app/assistant/service.py`、`tools.py`、`state.py` |
| **3. 证据检索** | Hybrid RAG：GraphRAG（Neo4j 子图）+ 向量检索（ChromaDB）+ 重排序 | `app/graphrag/`、`app/graph/neo4j_store.py`、`app/rag/` |
| **4. 资源生成** | 5 个 LangChain Agent（Document/Mindmap/Exercise/Code/VideoScript）并行；每类走「生成→校验→LLM 修复→再校验→降级保留原因」链路 | `app/agents/resource_agents.py`、`service.py` |
| **5. LLM JSON 解析** | 手动兜底解析，容忍模型返回的非严格 JSON | `app/core/manual_json.py` |
| **6. 练习评估** | 按题型分发评分：选择题规则匹配、简答/案例走 LLM rubric、编程题结构化评分（+AST 检查） | `app/exercises/service.py`（`_grade_attempt`） |
| **7. 诊断推荐** | 拓扑排序 + 优先级打分，返回逐节点 `reason/type/score/evidence` | `app/diagnosis/service.py`、`schemas.py` |
| **8. 学生画像** | 10 维度画像；LLM 抽取 + 行为增量更新 + 完整度计算；每个 weak_point/mastery 带证据来源 | `app/profile/`（extractor / behavior_updater / completeness） |
| **9. 成长追踪** | 时间轴事件关联练习/资源/反馈/画像；遗忘曲线预警 | `app/profile/timeline.py` |
| **10. 反馈闭环** | 负反馈触发重新解释/降难度/补例子/生成练习，并回显「系统做了什么调整」 | `app/assistant/feedback_analyzer.py` |
| **11. 服务单例** | ServiceCache 统一管理重资源服务（Neo4j/GraphRAG/Diagnosis/Extractor） | `app/core/service_cache.py` |
| **12. 配置** | Settings + 环境变量，`get_settings()` 统一取值 | `app/core/config.py` |
| **13. 前端可视化** | ECharts 画像图表、markmap 思维导图、Mermaid 流程图、KaTeX 公式 | `frontend/src/components/`、`views/` |

## 五、关键约束（AI 与人协作时必须遵守）

这些约定写在 `SKILL.md` 里，是保证「代码风格一致 + 闭环可修复」的核心：

**后端**
- 配置只通过 `Settings` + 环境变量，用 `get_settings()` 取，不硬编码
- 重资源服务走 `ServiceCache` 单例，不重复实例化
- 日志用 `logging.getLogger(__name__)`，**禁止 `print()`**
- `except Exception:` **必须记日志**，不允许静默吞异常
- 生产环境 `JWT_SECRET` 必须来自环境变量，不允许默认密钥
- ORM 用 SQLAlchemy 2.0 `Mapped` 风格
- 新路由：`routes/` 下建文件 → `router.py` 注册 → 前端 `api/` 封装

**前端**
- 按功能域用对应 Pinia store（auth/profile/assistant/learning）
- API 统一走 `api/` 封装 + `client.ts` 的 Axios 实例
- 类型定义放 `types/`，与后端 Schema 保持同步
- 组件按域分目录，样式用全局 CSS 变量 + `<style scoped>`

**改动流程约束**（脚手架式）
1. 改 Vue 前先读完整文件（注意 `watch` 从 `vue` 导入）
2. 改路由前确认 `deps.py` 有依赖注入函数
3. 改 ORM 前确认迁移/自动建表逻辑
4. 改 schemas 要同步更新前端 `types/`
5. 改评分逻辑集中在 `_grade_attempt` 按题型分发

## 六、"闭环修复 / 脚手架"是什么意思

- **自修复闭环（self-repair）**：资源生成不是「一次生成就交付」，而是 `生成 → 校验 → 若失败把原因喂回 LLM 修复 → 再校验 → 仍失败则标记失败并保留原因 + 支持单独重试`。这让不可控的 LLM 输出变得工程可靠。校验标准按资源类型定义（如 Document 正文 ≥200 字、Exercise 每题必须有题干+选项+答案）。
- **决策可解释闭环**：`enhance-system-intelligence` spec 的核心是让每个推荐/判断都带 `reason + evidence`，用户能看到「为什么系统这样判断」，反馈还能反向触发调整。
- **脚手架（scaffolding）**：指 `SKILL.md` + spec 模板 + 目录/命名/注册约定共同构成的「开发骨架」——新功能照着填即可，AI 和人都不容易走偏。
