# 系统可用化审查与改造路线

最后更新：2026-07-03

## 1. 审查结论

当前项目已经不是“只有基础界面和 demo 流程”的状态。代码中已经存在一条较完整的学习闭环：

```text
登录/注册
-> 学习主助手 LangGraph
-> 学生画像读取与更新
-> 语义记忆检索
-> GraphRAG 证据检索
-> 多 Agent 资源生成
-> 练习提交与练习记录
-> 错题与复习
-> 反馈写入
-> 行为画像更新
-> 成长时间轴与遗忘预警
```

因此后续文档不能再把这些能力笼统写成“待实现”。更准确的状态是：

- 多数核心模块已经有后端路由、数据库模型、服务层和前端页面。
- 最大风险不在“有没有功能”，而在“是否真实可联调、是否静默降级、是否可观测、是否所有 UI 都接入真实数据”。
- 下一阶段重点应从“补功能”转为“系统硬化、真实数据验证、可解释闭环和质量评测”。

## 3. 确实存在的问题

### 3.1 静默降级过多

当前系统有多处 fallback：

- LLM resolver fallback。
- GraphRAG fallback resolve。
- Neo4j property fallback path。
- 资源生成 fallback exercise。
- 画像抽取 deterministic fallback。
- demo memory seed。

这些 fallback 有些是合理的健壮性设计，但演示和测试时必须可见。

问题：

- 用户或评委可能不知道系统当前到底用了真实 LLM、真实 Neo4j、真实 Chroma，还是走了兜底。
- fallback 结果如果和真实链路混在一起，会让质量评估失真。

解决方向：

- 所有核心响应增加 `runtime_mode` 或 `degraded_reasons`。
- 前端调试面板显示：LLM、Neo4j、Chroma、course semantic index、memory index 的状态。
- 开发环境中对 fallback 给出明显 warning。
- 测试计划中区分“允许 fallback 的健壮性测试”和“不允许 fallback 的真实链路测试”。

### 3.2 健康检查不够完整

当前 `/health` 主要检查 Neo4j。

缺口：

- 没有暴露 LLM 配置状态。
- 没有暴露 embedding 配置状态。
- 没有暴露 Chroma collection count。
- 没有暴露课程语义索引维度是否匹配。
- 没有暴露 SQLite 初始化和关键表状态。

解决方向：

- 增加 `/api/admin/runtime-status` 或扩展 `/health`。
- 返回：
  - Neo4j connectivity
  - SQLite connectivity
  - LLM configured
  - embedding configured
  - memory collection count
  - course semantic collection count
  - active model names
  - degraded features

### 3.3 课程语义 RAG 尚未成为 canonical evidence 的入口

问题：

- `semantic_hits` 已进入 EvidencePackage，但命中结果还没有强制回查 Neo4j 并合并到 canonical evidence。
- 这会导致 semantic hits 更像“旁路调试信息”，而不是 GraphRAG 质量提升的核心环节。

解决方向：

- 实现 semantic hit 回灌。
- 合并到 `misconceptions`、`exercises`、`code_cases`、`document_chunks`。
- 保留 `semantic_hits` 作为解释轨迹。

### 3.4 前端仍有少量硬编码或展示不完整

已发现：

- `frontend/src/views/KnowledgeGraphView.vue` 中 `evidenceScore = computed(() => 0.6)` 仍是硬编码。
- 部分 fallback label 合理，但需要避免展示英文 enum。
- 练习历史和成长页面已存在，但需真实端到端操作验证。

解决方向：

- 图谱页 evidence score 接真实 GraphRAG evidence。
- EvidencePanel 补语义入口展示。
- 所有 `fallback`、`resolution_quality`、`target_type`、`view_type` 前端本地化。

### 3.5 时间轴事件来源仍不够完整

已有：

- profile update
- exercise result
- learning progress
- node mastery
- forgetting warning

缺口：

- assistant questions 未完全纳入。
- resource generation 未完全纳入。
- feedback milestone 未完全纳入。
- 复习救援事件还没有形成强闭环。

解决方向：

- 从 assistant messages 生成 `question_asked`。
- 从 resource generation records 生成 `resource_generated`。
- 从 assistant feedback 生成 `feedback_given`。
- 复习遗忘节点后生成 `review_rescue` 或类似里程碑。

### 3.6 测试仍偏“能编译/能启动”，缺少真实学生路径验收

需要建立真实用户路径测试：

```text
新学生注册/演示登录
-> 建立画像
-> 问一个具体问题
-> GraphRAG 返回真实 evidence
-> 生成资源
-> 做练习
-> 查看练习历史
-> 查看错题
-> 提交反馈
-> 查看画像变化
-> 查看成长时间轴
-> 查看学习路径更新
```

测试必须区分：

- 是否使用真实 Neo4j。
- 是否使用真实 LLM。
- 是否使用真实 embedding。
- 是否使用真实 SQLite 持久化。
- 是否有 demo seed 或 fallback 参与。

## 5. P1：系统体验和智能质量增强

### P1.1 反馈闭环更深接入主 Agent

当前反馈已经写入行为画像，但下一步要影响具体策略：

- 偏好代码：回答优先给代码。
- 觉得太难：降低抽象度，增加步骤。
- 觉得太简单：提高推导深度。
- 没看懂：下次同知识点先澄清误区。
- 想要例子：优先类比和具体案例。

验收：

- 同一学生连续反馈后，下次回答风格明显变化。
- 前端能看到“本次回答采用了哪些画像偏好”。

### P1.2 时间轴补全事件来源

补入：

- assistant question events。
- resource generation events。
- feedback milestone events。
- review rescue events。

验收：

- 学生能在成长页看到完整学习故事，而不只是画像变动。

### P1.3 学习路径和遗忘预警联动

当前诊断推荐和时间轴已经有基础逻辑。

增强：

- 遗忘节点在推荐中提权。
- 复习遗忘节点后解除预警。
- 触发复习成就或里程碑。
- 学习路径说明为什么把某个节点排前面。

### P1.4 GraphRAG EvidencePanel 可解释增强

需要展示：

- 中心知识点。
- 前置路径。
- 多跳依赖。
- 文档、练习、代码、FAQ 覆盖情况。
- semantic hits。
- missing evidence。
- ranking reason。
- resolution quality。

### P1.5 建立真实评测集

至少三类：

- 检索评测：学生问法 -> 期望知识点和证据类型。
- 学习路径评测：画像 -> 期望推荐节点。
- 闭环评测：练习/反馈 -> 画像变化。

## 7. 修订后的优先级判断

不应再把以下内容列为“未实现”：

- LangGraph 主助手。
- 反馈提交。
- 行为画像更新。
- 练习记录持久化。
- 错题本。
- 成长时间轴。
- 资源中心持久化。
- 学习路径基础推荐。

它们的真实状态是：

```text
已实现基础版本
-> 需要真实环境联调
-> 需要消除静默 fallback
-> 需要补可观测性
-> 需要补端到端测试
-> 需要增强个性化策略
```

当前最该做的不是再发散新增大模块，而是把已有系统打磨成可以经得住普通学生真实试用的闭环产品。
