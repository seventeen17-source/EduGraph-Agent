## 1. 排期原则

本项目时间紧，功能面广，排期不采用平均分配方式，而采用“先打通闭环，再逐步增强”的策略。

核心原则：

1. 6 月 10 日前完成最小可运行闭环的技术验证。
2. 6 月 18 日前完成主要功能开发。
3. 6 月 23 日前完成演示主流程。
4. 6 月 27 日前完成文档、PPT、视频初稿。
5. 6 月 28 日至 6 月 30 日只做修复、剪辑、包装和提交，不再大改架构。

功能优先级：

| 优先级 | 含义 | 示例 |
| --- | --- | --- |
| P0 | 必须完成，否则不满足赛题核心要求 | 对话式画像、5 类资源生成、多智能体流程、学习路径、知识库 |
| P1 | 强烈建议完成，用于拉开差距 | GraphRAG、知识图谱可视化、学习效果评估、防幻觉引用 |
| P2 | 有时间再做，用于冲奖增强 | 教师端、自动生成 PPT、视频真实渲染、复杂学习报告 |

## 5. 技术调整预案

### 5.1 图数据库调整

首选：Neo4j。

备选：

1. PostgreSQL 表结构模拟图谱。
2. JSON 文件维护课程图谱。
3. NetworkX 在后端内存中计算图路径。

调整条件：

1. Neo4j 安装部署影响进度。
2. 团队成员不熟悉 Cypher。
3. 演示环境不方便启动多个服务。

### 5.2 向量数据库调整

首选：Chroma 或 Milvus。

备选：

1. FAISS 本地索引。
2. LangChain InMemoryVectorStore。
3. 数据量较小时使用简单 BM25 + Embedding 混合检索。

调整条件：

1. 部署复杂度过高。
2. 检索数据规模较小。
3. 初赛演示更重视流程完整性。

### 5.3 多模态资源调整

首选：生成视频/动画脚本，并展示关键帧或分镜。

增强：接入多模态生成工具生成短视频片段。

调整条件：

1. 视频生成耗时或不稳定。
2. API 成本高。
3. 演示视频时长不足以展示完整生成过程。

### 5.4 Agent 编排调整

首选：LangGraph 状态图。

备选：

1. LangChain Runnable 串联。
2. FastAPI 中自定义状态机。
3. 对关键流程做固定编排，保留 Agent 日志展示。

调整条件：

1. LangGraph 调试成本高。
2. 多分支流程影响稳定性。
3. 演示阶段需要减少不可控行为。

## 8. 演示主线建议

最终演示建议围绕一个具体学生展开。

示例学生：

```text
学生：计算机专业大二学生
目标：学习人工智能导论，为课程项目做准备
基础：学过 Python 和高等数学，但机器学习基础薄弱
偏好：喜欢图解和代码案例
问题：不理解梯度下降和神经网络训练过程
```

演示流程：

1. 学生通过自然语言对话建立画像。
2. 系统抽取画像并显示 8 个维度。
3. 系统基于课程图谱定位“梯度下降”的前置知识。
4. GraphRAG 检索课程资料并给出引用。
5. 多智能体生成讲解文档、思维导图、练习题、视频脚本、代码案例。
6. 系统规划学习路径并解释推荐理由。
7. 学生完成练习，系统评估薄弱点。
8. 画像更新，学习路径自动调整。
