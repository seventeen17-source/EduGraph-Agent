# EduGraph-Agent 需求分析说明书

版本：v1.0

日期：2026-06-14


## 1. 文档目的

本文档用于说明 EduGraph-Agent 项目要做什么、为什么要做、给谁用、需要实现哪些功能，以及每个功能做到什么程度才算可验收。

- [A3 初步开发文档](../docs/project/A3_initial_development_doc.md)
- [项目进度台账](../docs/project/PROJECT_PROGRESS.md)

本文档不直接替代开发排期，也不记录每日进度。它主要回答一个问题：

> 我们要做一个怎样的系统，系统需要满足哪些需求？

## 2. 项目背景

本项目面向中国软件杯 A3 赛题：基于大模型的个性化资源生成与学习多智能体系统。

赛题希望参赛团队开发一个面向高校课程学习的智能系统。这个系统不能只是普通聊天机器人，而应该能根据学生的基础、目标、偏好和薄弱点，生成个性化学习资源，并持续规划学习路径。

我们选择的项目名称是 EduGraph-Agent。

项目核心定位是：

> 基于课程知识图谱和 GraphRAG 的个性化学习多智能体系统。

一句话解释：

> 学生告诉系统自己想学什么、哪里不会；系统根据课程知识图谱找到相关知识点和前置知识，再结合教材/讲义证据生成讲解、练习、代码案例、思维导图和视频脚本，并根据练习结果更新学生画像和学习路径。

## 3. 要解决的问题

### 3.1 新时代大学生的学习特征与需求

围绕高校《机器学习》课程的教学场景观察与赛题背景分析，新时代大学生的学习呈现以下特征，构成本系统的需求出发点：

| 特征 | 表现 | 对系统的需求 |
|------|------|-------------|
| 数字原住民，多模态偏好 | 习惯视频、图解、交互式内容，纯文字讲义吸收效率低 | 学习资源需多模态：图文、导图、代码、视频脚本、图片 |
| 生成式 AI 工具已成日常 | 普遍用通用聊天 AI 辅助学习，但通用 AI 不懂课程结构，"答了但没学会" | 需要懂课程知识体系、能追溯依据的专业学习智能体 |
| 基础差异显著 | 转专业、跨学科选课普遍；同一门课里数学/编程基础参差 | 需要个体画像与个性化讲解深度、资源形式适配 |
| 知识体系庞杂，规划能力不足 | 以本课程为例：11 章、50 个核心知识点、上百条前置依赖关系，"不知道先学什么"是普遍困境 | 需要基于依赖关系的自动学习路径规划 |
| 碎片化学习，易学易忘 | 学习时间零散，缺乏系统复习机制 | 需要遗忘检测、复习推荐与成长追踪 |
| 对反馈的即时性期待高 | 问完就走，传统系统不跟踪掌握情况 | 需要练习评估→画像更新→路径调整的闭环 |

### 3.2 现有方案的不足

1. **传统在线课程平台**：内容"千人一面"，无诊断、无路径、无跟踪；
2. **通用聊天 AI**：不感知课程知识结构与前置关系，容易直接给答案而跳过必要前置；无来源依据，存在幻觉风险；不记忆学生状态；
3. **题库/刷题类工具**：只覆盖"练"，与"学"和"规划"割裂。

### 3.3 本系统要解决的问题

- 让学生可以用自然语言表达学习需求。
- 让系统能识别学生真正涉及的知识点。
- 让系统能根据知识图谱补充前置知识和关联知识。
- 让系统基于可靠课程资料生成学习资源。
- 让系统通过练习和反馈更新学生画像。
- 让学习路径可以随学生掌握情况动态调整。

### 3.4 前沿 AI 技术与学习需求的结合点

| 学生需求痛点 | 采用的前沿 AI 技术 | 系统落地功能 |
|-------------|-------------------|-------------|
| 不知道先学什么、后学什么 | 知识图谱 + Kahn 拓扑排序诊断 | 学习路径页：拓扑有序推荐 + 逐节点理由 |
| 通用 AI 答疑无课程结构、易幻觉 | GraphRAG / HybridRAG（图检索 × 语义向量双通道） | EvidencePanel：多跳依赖路径 + 来源可追溯 |
| 统一材料不适配个人基础 | LLM 画像抽取 + 8 维学生画像 + 行为画像 EMA 建模 | 画像对话构建；讲解深度/形式随画像适配 |
| 偏好不同内容模态 | 多智能体并发生成 + 文生图（讯飞 HiDream）+ 结构化导图 | 一键生成 6 类学习资源 |
| 学完就忘、缺乏复习节奏 | 遗忘曲线检测 + ChromaDB 语义记忆 | 成长时间轴、遗忘预警、复习推荐 |
| 问完不跟踪、无学习闭环 | LangGraph 多智能体编排（17 节点）+ 练习评估回写 | 对话→练习→画像更新→路径调整完整闭环 |

## 4. 项目目标

### 4.1 总体目标

构建一个可以演示完整学习闭环的 Web 系统：

```text
学生画像构建 -> 知识诊断 -> GraphRAG 检索 -> 多智能体资源生成 -> 学习路径规划 -> 学习效果评估 -> 学生画像更新
```

### 4.2 MVP 目标

MVP 阶段至少要完成以下能力：

1. 学生可以通过对话输入个人基础、目标和偏好。
2. 系统可以形成结构化学生画像。
3. 系统内置一门完整课程知识库，当前选定为机器学习。
4. 系统可以根据问题定位知识点，例如"反向传播""梯度下降""K-Means"。
5. 系统可以返回知识点的前置关系、相关知识、讲义片段、题目和来源。
6. 系统可以生成至少 5 类学习资源：
   - 讲解文档
   - 思维导图
   - 练习题
   - 视频/动画脚本
   - 代码案例
7. 系统可以给出个性化学习路径。
8. 系统可以根据练习结果更新学生薄弱点或掌握状态。
9. 系统可以展示引用来源和图谱依据，降低大模型幻觉风险。

### 4.3 非目标

当前阶段不追求：

1. 覆盖多门课程。
2. 完整教师后台。
3. 真实视频自动生成。
4. 大规模用户并发。
5. 全自动从 PDF 抽取知识图谱。
6. 非机器学习课程的泛化能力。

这些可以作为后续扩展，而不是当前 MVP 的主线。

## 5. 用户角色

### 5.1 学生

学生是系统的主要使用者。

学生可以：

- 通过对话描述自己的学习基础和学习目标。
- 查询某个知识点或提出自然语言问题。
- 查看系统生成的讲解、图、题目、代码案例和视频脚本。
- 查看推荐学习路径。
- 完成练习题。
- 查看自己的掌握情况和薄弱点变化。

典型学生示例：

```text
计算机专业大二学生，学过 Python 和高等数学，但机器学习基础薄弱。
他想为了课程项目学习机器学习，偏好图解和代码案例。
他提出问题："神经网络训练时那个从后往前算梯度的过程是什么？"
```

系统应能识别这个问题对应"反向传播"，并补充梯度下降、多层神经网络、Mini-batch 等前置或关联内容。

### 5.2 教师或课程维护者

教师或课程维护者不是当前 MVP 的主要操作对象，但需要在系统设计中预留。

他们未来可以：

- 维护课程章节。
- 维护知识点。
- 审核或修正知识图谱关系。
- 审核生成的学习资源。
- 查看学生整体学习情况。

当前 MVP 中，可以先通过静态 JSON、Markdown 和 Neo4j 导入脚本完成课程维护。

### 5.3 系统管理员

管理员负责系统配置和运行维护。

管理员可以：

- 配置大模型 API。
- 配置 Neo4j 数据库。
- 导入课程知识库。
- 查看服务状态和日志。

当前 MVP 中，管理员功能可以先通过配置文件、命令行脚本和后端接口完成，不需要完整后台页面。

## 6. 核心业务流程

### 6.1 首次使用流程

1. 学生进入系统。
2. 系统询问学生的专业背景、学习目标、基础和偏好。
3. 学生用自然语言回答。
4. Profile Agent 抽取学生画像字段。
5. 前端展示学生画像卡片。
6. 系统根据画像和课程图谱推荐初始学习路径。

输入示例：

```text
我是计算机专业大二学生，学过 Python 和高数，但是机器学习基础比较弱。
我想为了课程项目学懂机器学习，比较喜欢图解和代码案例。
```

输出应包含：

- 专业背景：计算机专业、大二
- 学习目标：课程项目
- 已有基础：Python、高等数学
- 薄弱点：机器学习基础
- 学习偏好：图解、代码案例

### 6.2 学生提问流程

1. 学生输入问题。
2. Node Resolver 将自然语言问题映射到知识点。
3. Knowledge Agent 查询知识图谱。
4. Retrieval Agent 组装证据包。
5. Resource Agents 根据证据包生成资源。
6. 前端展示知识点、前置路径、讲义证据和生成内容。

输入示例：

```text
神经网络训练时那个从后往前算梯度的过程是什么？
```

系统应输出：

- 识别知识点：反向传播
- 前置知识：梯度下降、Mini-batch、多层神经网络
- 相关知识：CNN、RNN、Transformer
- 讲义片段：来自 ch06 神经网络与深度学习基础
- 练习推荐：反向传播和梯度下降分工、ReLU 反向梯度等

### 6.3 学习路径规划流程

1. 系统读取学生画像。
2. 系统读取学生目标知识点。
3. 图谱查询该知识点的前置节点和后续节点。
4. 系统过滤学生已掌握内容。
5. 系统按难度和依赖关系生成路径。
6. 前端以路径图或步骤列表展示。

示例：

```text
线性回归 -> 损失函数 -> 梯度下降 -> 多层神经网络 -> 反向传播
```

每一步都需要说明推荐理由，例如：

- "梯度下降是反向传播计算出梯度之后常用的参数更新方法。"
- "多层神经网络是理解反向传播逐层传递梯度的结构基础。"

### 6.4 学习效果评估流程

1. 学生完成练习题。
2. Evaluation Agent 判断答案是否正确。
3. 系统根据题目关联知识点更新掌握度。
4. 如果答错，系统记录薄弱点和常见误区。
5. Planning Agent 调整后续学习路径。

示例：

学生把"反向传播"等同于"梯度下降"，系统应提示：

```text
反向传播负责计算各层参数的梯度；
梯度下降负责使用这些梯度更新参数。
```

然后系统应将"反向传播"和"梯度下降的关系"标记为需要复习。

## 7. 功能需求

### 7.1 学生画像模块

#### 功能说明

系统需要通过对话收集学生信息，并形成结构化画像。

#### 必须包含的画像维度

1. 专业背景：专业、年级、已学课程。
2. 学习目标：考试、课程项目、竞赛、科研、就业或兴趣。
3. 知识基础：已掌握和未掌握的主题。
4. 学习进度：当前学习章节、已完成知识点。
5. 认知风格：偏理论、偏案例、偏代码、偏图解。
6. 薄弱点：自述薄弱点和系统诊断薄弱点。
7. 学习偏好：文档、图解、练习、视频脚本、代码案例。
8. 能力状态：编程能力、数学基础、阅读能力、综合应用能力。

#### 验收标准

- 能从一段自然语言中抽取至少 4 个画像维度。
- 能以 JSON 形式保存画像。
- 能在前端或接口响应中展示画像字段。
- 能在学生答题后更新薄弱点。

### 7.2 课程知识库模块

#### 功能说明

系统需要内置一门完整高校课程的知识库。当前课程为机器学习。

#### 数据范围

当前课程结构包含 11 章，其中 P0 优先覆盖 ch01 到 ch06：

- ch01 机器学习导论与模型评估
- ch02 数学基础与优化基础
- ch03 线性模型与梯度优化
- ch04 经典监督学习算法
- ch05 无监督学习与特征表示
- ch06 神经网络与深度学习基础
- ch07 计算学习理论
- ch08 半监督学习
- ch09 概率图模型
- ch10 规则学习
- ch11 强化学习

#### 验收标准

- 至少有 1 门课程。
- 至少有 10 个章节结构。
- 至少有 50 个知识点。
- P0 章节每个核心知识点至少有一个讲义 chunk。
- 每个 chunk 必须绑定知识点和来源。

### 7.3 课程知识图谱模块

#### 功能说明

知识图谱用于表达课程结构和知识点依赖关系。

#### 节点类型

- Course：课程
- Chapter：章节
- KnowledgePoint：知识点
- DocumentChunk：讲义片段
- Exercise：练习题
- CodeCase：代码案例
- Source：参考来源
- FAQ 或 Misconception：常见问题或误区

#### 关系类型

- HAS_CHAPTER：课程包含章节
- CONTAINS：章节包含知识点
- PREREQUISITE：知识点前置关系
- RELATED：知识点关联关系
- SUPPORTS：讲义片段支持知识点
- ASSESSES：题目测评知识点
- PRACTICES：代码案例练习知识点
- CITES_SOURCE：内容引用来源

#### 验收标准

- Neo4j 能导入完整图谱数据。
- 查询某个知识点时，能返回前置知识和相关知识。
- 查询某个知识点时，能返回支撑讲义和练习题。
- 图谱边方向必须清晰，尤其是 PREREQUISITE。

### 7.4 GraphRAG 检索模块

#### 功能说明

GraphRAG 不是普通全文检索。它需要先定位知识点，再根据图谱扩展上下文，最后检索相关讲义证据。

#### 基本流程

1. 理解学生问题。
2. 定位中心知识点。
3. 查询前置知识、相关知识和后续知识。
4. 找到这些知识点对应的讲义 chunk。
5. 组装 Evidence Package。
6. 返回给资源生成 Agent。

#### Evidence Package 至少包含

- 中心知识点
- 前置知识点
- 相关知识点
- 讲义片段
- 练习题
- 代码案例
- 来源信息
- 不确定性说明

#### 验收标准

- 输入自然语言问题，可以定位到正确知识点。
- 对 `ml_gradient_descent`、`ml_kmeans`、`ml_backpropagation` 至少三个节点能返回完整证据包。
- 返回结果中必须有来源或证据说明。
- 当存在多个候选知识点时，需要返回不确定性信息。

### 7.5 多智能体资源生成模块

#### 功能说明

系统需要把 GraphRAG 证据包交给不同资源生成 Agent，生成个性化学习材料。

#### 至少 5 类资源

1. 讲解文档 Agent：生成通俗解释。
2. 思维导图 Agent：生成 Mermaid 或 JSON 结构。
3. 练习题 Agent：生成选择题、简答题或代码题。
4. 视频脚本 Agent：生成讲解脚本和分镜。
5. 代码案例 Agent：生成 Python 示例或练习。

#### 个性化要求

资源生成需要参考学生画像：

- 如果学生偏图解，优先生成图示和路径。
- 如果学生偏代码，优先给代码案例。
- 如果学生基础薄弱，解释要降低抽象程度。
- 如果学生已经掌握前置知识，可以减少基础铺垫。

#### 验收标准

- 至少能对一个知识点生成 5 类资源。
- 生成内容必须引用 Evidence Package 中的知识点或讲义依据。
- 生成内容不能只是一段泛泛回答，要体现学生画像。
- 代码案例需要可读、可运行或至少语法合理。

### 7.6 学习路径规划模块

#### 功能说明

系统需要根据知识图谱和学生画像生成学习路径。

#### 路径规划依据

- 目标知识点
- 前置知识关系
- 知识点难度
- 学生已掌握知识
- 学生薄弱点
- 学生学习目标

#### 验收标准

- 能生成从前置知识到目标知识点的路径。
- 能说明每个推荐节点的理由。
- 能跳过学生已掌握节点，或标记为快速复习。
- 练习后能根据结果调整路径。

### 7.7 学习评估模块

#### 功能说明

系统需要通过练习题评估学生掌握情况。

#### 题目类型

- choice：选择题
- short_answer：简答题
- coding：代码题
- case_analysis：案例分析题

#### 验收标准

- 能根据选择题答案自动判断正误。
- 能根据 coding 题测试用例判断结果。
- 能根据题目关联知识点更新掌握状态。
- 答错时能给出针对性反馈，而不是只说"错误"。

### 7.8 解释性与防幻觉模块

#### 功能说明

系统生成内容需要尽量可追溯，避免大模型脱离课程资料胡编。

#### 必须支持

- 显示引用来源。
- 显示关联知识点。
- 显示推荐路径理由。
- 对不确定结果给出提示。
- 对生成内容做基本质量检查。

#### 验收标准

- GraphRAG 响应中包含 sources。
- 资源生成时能说明依据了哪些知识点或 chunk。
- 对找不到明确知识点的问题，不强行给出唯一结论。

## 8. 数据需求

### 8.1 课程数据

课程数据主要存放在：

- `data/course/course_meta.json`
- `data/course/chapters.json`
- `data/course/knowledge_points.json`
- `data/course/graph_edges.json`

课程数据需要表达：

- 课程名称
- 章节结构
- 知识点
- 知识点前置关系
- 知识点关联关系
- 难度
- 学习目标

### 8.2 讲义数据

讲义数据主要存放在：

- `data/docs/`

讲义 chunk 至少需要包含：

- `chunk_id`
- `title`
- `chapter_id`
- `knowledge_node_ids`
- `source_ids`
- `content`

注意：讲义内容应是自写总结，不能大段复制教材原文。

### 8.3 题库数据

题库数据主要存放在：

- `data/exercises/exercises_index.json`
- `data/exercises/chxx_*.json`

每道题至少需要包含：

- 题目 ID
- 题目类型
- 关联知识点
- 难度
- 问题正文
- 答案或评分规则
- 自适应反馈

### 8.4 来源数据

来源数据主要存放在：

- `data/sources/sources.json`

来源包括：

- 本地教材/PDF
- 开放课程
- 官方文档

每个讲义 chunk、题目和代码案例都应该尽量绑定 `source_ids`。

### 8.5 学生画像数据

学生画像建议结构化保存，核心字段包括：

- background
- learning_goal
- knowledge_base
- progress
- cognitive_style
- weak_points
- preferences
- ability_state
- update_history

每个画像字段最好保存来源和置信度，例如来自第几轮对话或哪次练习结果。

## 11. 非功能需求

### 11.1 可用性

- 新队友应能通过 README 和文档快速启动项目。
- 演示流程必须稳定，不依赖临时手工操作。
- 接口错误时应返回清晰错误信息。

### 11.2 可解释性

- 系统推荐必须说明理由。
- GraphRAG 结果必须包含知识点关系或来源。
- 学习路径不能只给列表，要说明为什么这么排。

### 11.3 可维护性

- 课程数据、题库、讲义、来源应分文件维护。
- 知识点 ID 命名要稳定。
- 导入脚本需要可重复执行。

### 11.4 安全性

- API Key 不能提交到 Git。
- 本地教材 PDF 不提交到 Git 仓库。
- 生成内容不能大段复制教材原文。

### 11.5 性能

MVP 阶段性能要求不高，但应满足：

- 单次 GraphRAG 查询在演示环境中能可接受地返回。
- Neo4j 查询不应阻塞整个服务。
- 前端页面切换和基础渲染流畅。

## 16. 常见概念解释

### 学生画像

学生画像就是系统对学生情况的结构化理解，例如专业、目标、基础、偏好、薄弱点等。

### 知识图谱

知识图谱就是把课程知识点和它们之间的关系组织成图。例如"反向传播"需要先理解"多层神经网络"和"梯度下降"。

### GraphRAG

GraphRAG 是图谱增强检索。普通 RAG 主要从文档中找相似片段；GraphRAG 会先根据知识图谱确定问题属于哪个知识点，再找前置知识、相关知识和对应讲义证据。

### 多智能体

多智能体不是多个聊天窗口，而是把任务拆成多个角色。例如画像 Agent 负责理解学生，检索 Agent 负责找证据，练习 Agent 负责出题，评估 Agent 负责判断学习效果。

### Evidence Package

Evidence Package 是系统给生成 Agent 的证据包，里面包括中心知识点、前置知识、讲义片段、题目、代码案例和来源。它的作用是让生成内容有依据。

## 17. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 功能过多做不完 | 页面、Agent、图谱、生成都要做 | 先保证 MVP 闭环，再做增强功能 |
| 生成内容不可靠 | 解释错误或编造来源 | 强制使用 GraphRAG 证据包和来源 |
| 知识库质量不够 | 检索不到有效内容 | 优先补 P0 章节核心知识点 |
| 前端展示不清楚 | 评委看不出系统价值 | 重点展示画像、图谱、路径、证据和资源 |
| Agent 过于抽象 | 看起来只是多个 Prompt | 展示每个 Agent 的输入、输出和分工 |
| 演示不稳定 | 现场接口失败或数据缺失 | 固定演示学生、演示问题和核心节点 |
